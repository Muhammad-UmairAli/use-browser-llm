# Task: Real-browser Comlink-proxy integration test

## Description

The existing real-browser integration test (`test/integration/worker-boundary.browser.test.ts`) only calls `checkCache()` — a plain scalar round-trip. It never exercises `Comlink.proxy()`, the mechanism `loadModel`'s progress callback and `streamGenerate`'s token callback actually depend on in production. Add a new test that genuinely exercises a proxied callback over a real `postMessage` boundary.

## Acceptance Criteria
- [ ] New minimal test-fixture worker (e.g. `test/integration/fixtures/echo-worker.ts`) exposes one method taking a proxied callback (e.g. `streamTokens(count, onToken)`) that invokes it N times then resolves — no `MLCEngine`/WebGPU dependency
- [ ] New real-browser test (`@vitest/browser` + Playwright, same setup as the existing `checkCache()` test) wraps the fixture worker with `Comlink.wrap()`, passes a `Comlink.proxy()`-wrapped callback, and asserts it fires the expected number of times, in order, before the owning call resolves
- [ ] Test runs under the existing `pnpm test:integration` script and CI job — no new tooling/config needed
- [ ] Existing `checkCache()` integration test is unaffected

## Technical Details

Deliberately does NOT use the real `worker.ts`/`MLCEngine` — this task tests Comlink's proxy mechanism itself, which needs no WebGPU or model. Mirrors the existing test's structure (`test/integration/worker-boundary.browser.test.ts`) for consistency.
