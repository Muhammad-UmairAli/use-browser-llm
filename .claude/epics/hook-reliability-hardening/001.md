# Task: Generation-phase crash/watchdog coverage

## Description

Extend the existing load-phase crash/watchdog mechanism to `generate()` and `streamGenerate()`. Currently a worker crash or silent browser OOM-kill mid-generation hangs forever — `EngineClient.onCrash()` (Phase 1) is only subscribed during the load effect.

## Acceptance Criteria
- [ ] `generate()` subscribes to `onCrash()` for the duration of the call; a crash rejects the pending promise with `WorkerCrashError` and clears `isGenerating`
- [ ] `generate()` has a flat, configurable timeout (conservative default, e.g. 60-120s) that rejects with `WorkerCrashError` and terminates the client if no resolution occurs — no intermediate signal exists for a single non-streaming call, unlike the load phase's progress ticks
- [ ] `streamGenerate()` subscribes to `onCrash()` for the duration of the generator's lifecycle (entry to `finally`); a crash surfaces as a thrown `WorkerCrashError` from the async generator
- [ ] `streamGenerate()` has an inactivity watchdog reset on every `onToken` call (mirrors the load-phase reset-on-progress pattern), not just on the outer promise settling
- [ ] Every new path has a dedicated mocked-engine test that demonstrably fails without the fix (verified during review, not just written)

## Technical Details

Reuses `EngineClient.onCrash()` (`src/engine-client.ts`, unchanged) — no new crash-detection mechanism. Follows the load-phase watchdog's termination-before-dispatch ordering (`src/use-browser-llm.ts`'s existing `resetWatchdog`/`clearWatchdog` pattern) to avoid a late-resolution flap after a timeout fires.

## Dependencies

None — extends existing Phase 1 infrastructure directly. Conflicts with task 003 (same file, `src/use-browser-llm.ts`) — do not run concurrently.
