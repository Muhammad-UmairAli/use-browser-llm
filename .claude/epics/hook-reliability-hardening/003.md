# Task: statusRef render-safety fix

## Description

`statusRef.current = state.status` is currently mirrored directly in `useLocalLLM`'s (now `useBrowserLLM`'s) render body, not inside an effect. This is safe under the hook's current usage but not strictly render-safe under React 18 concurrent features in general — a discarded/prerendered render pass could in principle write a status the committed tree never shows.

## Acceptance Criteria
- [ ] `statusRef.current` is only ever written from a committed render (`useLayoutEffect`), never directly in the render body
- [ ] No behavioral change to `generate()`/`streamGenerate()`/`abort()`'s status-gating logic — this is purely a render-safety fix, not a functional change
- [ ] Existing 47-test suite passes unchanged (no new test strictly required for this fix alone, since it has no observable behavior difference under the hook's current usage, but note in the PR why)

## Technical Details

Minimal fix: move the assignment into `useLayoutEffect(() => { statusRef.current = state.status; }, [state.status])`. `useSyncExternalStore` was considered and rejected as overkill (no external subscribers, single internal ref, three stable consumers) — see epic.md's Architecture Decisions.

## Dependencies

None — conflicts with task 001 (same file, `src/use-browser-llm.ts`) — do not run concurrently.
